# ch3 - debugging cpp programs

### 3.1 - syntax and semantic errors
- errors usually fall into one of two categories: syntax errors and semantic errors
- a **syntax error** is when you write a statement that is not valid according to the grammar of the cpp language
- a **semantic error** is an error in meaning, which means although syntactically valid, it violates other rules of the language or just does not do what the programmer intended to do

### 3.2 - the debugging process
- debugging the problem consists of 6 steps:
    1. find the root cause
    2. understand the problem
    3. determine a fix
    4. repair the issue
    5. retest for fix 
    6. retest to check for new problems

### 3.3 - a strategy for debugging
- finding the problems by running the program: 
  1. figure out how to reproduce the problem
  2. run the program to narrow down where the problem is 
  3. repeat until you find the program 
- **reproduction steps** are a list of clear and precise steps that can be followed to cause an issue to recur with a high level of predictability

### 3.4 - basic debugging tactics
- tactic 1: commenting out your code
- tactic 2: validating your code flow
  - tip: when you're printing information for debugging processes, use `std::cerr` instead of `std::cout` (one reason for this is that `std::cout` is buffered, while `std::cerr` is unbuffered)
- tactic 3: printing the values (but generally not recommended because it clutters your code)

### 3.5 - more debugging tactics 
- when you print the values to debug your code using `std::cerr`, it becomes a hassle to remove them all or reuse them later, so a way to get through this is to use preprocessor directives to enable and disable the debug statements at the time of compilation using `#define` and `#ifdef`
- another approach to debugging efficiently is to use a logger
- a log is a sequence of events that have happened, usually time-stamped

### 3.6 - stepping

- purpose: pause your actual running program and watch it execute one line at a time, instead of guessing what it does or littering code with std::cout to check values
- start a debug session with the bug icon (not the green run arrow); this compiles with extra debug info the debugger needs, then runs the program under debugger control
- step into (F7): runs the current line; if it's a function call, jumps inside that function so you can watch its internals line by line too. use this when you want to see exactly what a called function is doing
- step over (F8): runs the current line; if it's a function call, runs that entire function to completion without pausing inside it, then stops on the next line after the call. use this when you trust the function works and just want to move past it
- step out (shift+F8): if you're currently paused inside a function (e.g. because you stepped into it), this finishes running the rest of that function automatically and pauses you right where it returns to, back in the caller
- run to cursor (alt+F9): click any later line and the program runs at full speed until it reaches that line, then pauses. faster than manually stepping through a long stretch you don't care about
- overall idea: stepping turns "i think the bug is somewhere in here" into "i can see exactly where things go wrong," by literally watching execution happen one instruction at a time

### 3.7 - running and breakpoints

- problem stepping alone doesn't solve: if the part you care about is hundreds of lines in, stepping one line at a time to get there is painfully slow. breakpoints fix this
- breakpoint = a marker on a specific line that tells the debugger "run at full normal speed until you reach this exact line, then pause automatically"
- set one by clicking in the gutter (thin strip left of the line numbers) next to the line you want to stop at; a red dot appears
- must run with debug (not the normal run button) for breakpoints to actually trigger
- once paused at a breakpoint, all the stepping tools from 3.6 (step into/over/out) become available to keep investigating from there
- resume/continue (F9): runs at full speed again, either finishing the program or stopping at the next breakpoint if more exist
- multiple breakpoints allowed across different lines/files; debugger stops at whichever it reaches first
- click a breakpoint dot again to remove it entirely
- right-click a breakpoint for more options: disable temporarily without deleting, or (clion-specific) add a condition so it only pauses when something specific is true (e.g. only pause when x == 5, useful for not stopping on every loop iteration)

### 3.8 - watching variables

- problem: breakpoints and stepping get you to the right spot in the code, but you still need to actually see what your variables hold once you're there
- variables panel (debug panel at the bottom, "variables" tab): automatically lists every variable currently in scope at your paused location, with live values that update in real time as you step forward
- this beats std::cout debugging because you see everything in scope continuously, without writing extra print lines or deciding in advance what to check
- watches: for tracking a specific variable or expression even as you move between functions/scopes where it would otherwise disappear from the normal variables list. right-click a variable in code, add to watches, or manually type an expression into the watches section
- hovering your mouse over a variable name in the code (while paused) shows a quick popup with its current value, no need to open any panel for a fast check
- some debuggers (including clion) let you actually edit a variable's value mid-pause, to test "what if this were different" without restarting the whole program

### 3.9 - the call stack

- problem: in real programs functions call other functions which call other functions. if you're paused deep inside one, it's easy to lose track of how you even got there
- call stack = the ordered list of every function currently "in progress" (called, but not yet finished/returned). whichever function you're paused inside sits at the top, the function that called it is next below, and so on down to main() at the bottom
- example: if bar() is called by foo() which is called by main(), and you're paused inside bar(), the stack reads top to bottom: bar(), foo(), main()
- in clion this is the debug panel's "frames" tab; each entry is one function call in progress, called a "stack frame," with its own local variables/parameters as they were at that point
- click any frame in the list to instantly jump to that exact spot in the code, with the variables panel updating to show that function's state at that moment, no need to manually step backward to see it
- why it matters: if a bug is in a utility function called from many different places, just knowing "the bug is in this function" isn't enough, you need to know which specific call chain led to it and what the state was at each level. the call stack gives you that whole chain in one view, instead of re-running and re-stepping the whole program from scratch to retrace it
